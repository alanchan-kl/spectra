The Idea

Four kinds of testing. One framework.

Most teams stitch together separate, often paid tools for each test type. Spectra unifies them under a single monorepo, a single config model, and a single reporting story — at zero licensing cost.

  • Consistent — the same env/system switching and data factories across every layer.
  • Portable — everything runs in Docker; no machine-specific tool sprawl.
  • Observable — functional results in Allure, performance in live Grafana + offline reports.
  • CI-ready — thresholds and exit codes gate the GitHub Actions pipeline.

Architecture

A clean monorepo

🌐@spectra/web

Playwright + BDD. Gherkin features, step definitions, page objects, API request specs.

📱@spectra/mobile

Maestro YAML flows for iOS & Android, with reusable sub-flows. Runs on an emulator.

⚡@spectra/performance

k6 scenarios (smoke/load/stress/breakpoint/soak), thresholds, Docker + Grafana.

🧩@spectra/shared

One config resolver, Faker data factories, and types shared across every package.

Switch target with env vars: SYSTEM, ENV, PLATFORM — one codebase, many environments.

Performance Engineering

Five test types — each answers a different question

TypeProfileQuestion it answers
Smoke1 VU, 1 iterationDoes it work at all?
Loadramp to expected peakOK under normal traffic?
Stresspush past peakWhere does it degrade?
Breakpointrising rate until SLA breaksWhat's the capacity ceiling?
Soaksteady load held for hoursDoes it stay healthy over time? (leaks)

Thresholds (e.g. p(95)<500ms, error rate<1%) are the pass/fail gate — a breach makes k6 exit non-zero and fails CI.

Containerised

Run anything — install almost nothing

k6 runs as a container, so a contributor needs only Docker — no native k6, no Node toolchain to execute tests. Two of the commands are "dry runs" that never touch the target.

# dry run — validates the script, sends 0 requests
pnpm --filter @spectra/performance docker:inspect

# minimal real run — 1 request
pnpm --filter @spectra/performance docker:smoke

# full load test, streamed live to Grafana
pnpm --filter @spectra/performance docker:load:mock
Engineering Decision

Never load-test someone else's API

Pointing thousands of requests at a public/demo API is abuse. So the stack ships a tiny local mock (nginx) that mirrors the endpoints under test.

  • Real curves — generate genuine load shapes locally.
  • Zero blast radius — nothing leaves your machine.
  • One flag — BASE_URL override points k6 anywhere.
# k6 → local mock, not a public API
docker compose run --rm \
  -e BASE_URL=http://mockapi \
  k6 run src/scenarios/load.ts

Reporting

Three ways to read results

🖥️Terminal

Instant text summary with threshold pass/fail at the end of every run.

📄Offline report

Each run drops report.json + report.html into its own timestamped folder — open in any browser, share as a file.

📈Live Grafana

Real-time time-series for demos and capacity analysis.

The HTML report is generated by k6's own handleSummary() hook — real measured data, no hand-editing.

Problems I Solved

The interesting engineering

TypeScript on k6 v2

Scenarios failed to load — k6's loader won't auto-resolve extensions. Fixed with explicit .ts imports + allowImportingTsExtensions.

Container networking

k6 → InfluxDB had to target the peer container hostname, not localhost; mounted the package so relative imports resolve.

Zero-touch dashboards

The popular community dashboard silently wouldn't bind. Replaced it with an auto-provisioned datasource + dashboard.

Per-run report folders

k6 can't mkdir at summary time, so an entrypoint wrapper creates reports/<scenario>-<ts>/ first.
